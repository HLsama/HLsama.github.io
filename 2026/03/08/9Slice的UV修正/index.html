9Slice的UV修正

2026/3/8 技术 UI

给到的需求是想要在使用9Slice功能的同时使用重复纹理，但由于使用9Slice后uv发生了变化，导致uv采样出现拉伸等异常，下文逐渐解释解决异常的原因以及解决方案。

unity版本：Unity6000.062f1

1.9Slice的UV异常

正常来说想要在一个UI后面叠加一个重复纹理是一件非常简单的事情，只需直接采样这个重复纹理再根据想要的方式对原图进行叠加就可以了。而9Slice则会导致重复纹理的采样出现拉伸挤压等情况：

究其原因是使用了9Slice后UV被不同程度的拉伸了。下面是开启9Slice前后UV的对比：

开启前：

开启后：

可以看到原本的UV根据9Slice的形状被挤压到了4个角，这就是最终重复纹理被挤压的元凶。或许有人会想：我们可以找到unity将源UV变换到9SliceUV的计算方式，然后再算出来个逆运算，或许就可以还原我们想要的UV。不过答案却不是这样，unity并没有对原生UV进行任何处理，导致UV被压缩的原因其实是在使用了9Slice之后UI的网格被拉伸和压缩了：

这是原生的UI网格：

这是使用9Slice的网格：

这就是9Slice的本质，导致压缩的真正元凶，由于实际上是网格的变化，unity并没对UV做任何手脚，我们也就找不到逆运算的方式去还原UV。

2.UV修正

如前文所述，其实我们并不可能修复原生UV，而UI的网格也没有UV2之类的可供我们选择，那么我们所剩下的只有一个了：屏幕UV。而屏幕uv是相对于整个屏幕不动的，用它采样的纹理会固定在屏幕中，也就不会根据我们物体的移动而移动，这并不是我们想要的：

对此有以下解决思路：我们可以根据物体在屏幕空间的位置，去整体移动整个屏幕UV，这样纹理就好像跟物体一起走一样了。

shader：

half4 Element = SAMPLE_TEXTURE2D(_ElementTex, sampler_ElementTex, (SSUV+float2(-_SrcPos.x,_SrcPos.y))*_ElementTex_ST.xy+_ElementTex_ST.zw);

在上述代码中，_SrcPos就是物体在屏幕空间中的位置，由于unity的屏幕空间坐标原点在左上角而通常物体的UV原点在左下角，所以对于u轴与v轴的处理方式是正相反的。

屏幕UV：

而_SrcPos是用C#传递过来的：先拿到物体的RectTransform，他所储存的内容是世界空间的点，unity提供了 RectTransformUtility.WorldToScreenPoint函数将其变换到屏幕空间，此外，我们还需要手动除以屏幕的宽和高。相关代码如下：

C#

Vector2 screenPos = RectTransformUtility.WorldToScreenPoint(null, myRectTransform.position);
Vector2 screenUV = new Vector2(screenPos.x / Screen.width, screenPos.y / Screen.height);

至此功能就完成了，目前还是有些缺点，例如由于这个方式是基于屏幕空间的，这导致Scene视角与Game视角下的内容不一致，Scene视角下是错误的内容，所以只能通过Game视角来还原。或许最合适的方式还是要还原原本的uv，根据网格的拉伸等比例等内容吧。暂时就先以屏幕空间的方式解决了。
