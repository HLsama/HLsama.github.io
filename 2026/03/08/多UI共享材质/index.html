多UI共享材质

2026/3/8 技术 UI

给本篇的需求本是跟9Slice的UV异常一同提出的需求，但比较重要所以额外拆分出来了。关于多物体共享材质其实有两种方式，其一是MaterialPropertyBlock，其二则是IMaterialModifier。虽然后者才是本次需求的解决方案，不过既然作为总结性质的文档，我会将两者都详细的讲一讲。

1.MaterialPropertyBlock

MaterialPropertyBlock通常是配合GPUInstancing来使用的。GPUInstancing可以大规模的渲染相同物品，如果每个物体的材质有所不同的时候，为每个材质建立材质球是个相当大的消耗，所以一般会使用MaterialPropertyBlock进行渲染。他可以让所有将要渲染的物体共享同一个材质球，但每个物体却可以是单独的材质参数，具体来说就是在对象每次在渲染的时候，Renderer就会把MaterialPropertyBlock中的参数覆盖掉原先材质对象中的相关数据，相比于整个材质球储存的信息，MaterialPropertyBlock只存储需要修改的几个变量，相当的轻量化。需要注意的是，在SRP中unity已经不太推荐使用MaterialPropertyBlock了（因为会导致SRP-Batcher被打断），而更推荐去为每个物体单独的设置材质。下面是unity官方手册中对MaterialPropertyBlock的说明：

Unity - Scripting API: MaterialPropertyBlock

官方示例代码：

using UnityEngine;

public class CreateTenGameObjects : MonoBehaviour
{
    public GameObject myPrefab;

    void Start()
    {
        // Loop through 10 GameObjects
        for (int i = 0; i < 10; i++)
        {
            // Instantiate a new GameObject at a unique position
            GameObject newObject = Instantiate(myPrefab, new Vector3(i * 2f, 0, 0), Quaternion.identity);

            // Create a new MaterialPropertyBlock
            MaterialPropertyBlock propertyBlock = new MaterialPropertyBlock();

            // Set a random color in the MaterialPropertyBlock
            propertyBlock.SetColor("_Color", Random.ColorHSV());

            // Apply the MaterialPropertyBlock to the GameObject
            newObject.GetComponent<MeshRenderer>().SetPropertyBlock(propertyBlock);
        }
    }
}

可以看到：MaterialPropertyBlock是依赖MeshRenderer来使用的，而UI则与常规的物体渲染有些不同，所以并没法使用这种东西。

2.IMaterialModifier

2.1.IMaterialModifier介绍

前文说过，UI渲染并不是MeshRenderer，而是，CanvasRenderer。那么UI真的不能共享同一个材质球吗，其实可以从unity的默认UIshader中找到答案。我们在使用UI系统时如果不填入材质球的话，unity会自动添加一个默认的材质球进行渲染，这个材质球只有一个却可以读取每个Image组件中的贴图，为同一材质球的两次渲染设置不同的贴图。

2.2.使用方式

我们需要构建一个脚本来控制我们材质参数，他将继承自IMaterialModifier，而想要继承它，我们必须实现一个GetModifiedMaterial方法。此外，与其他脚本不同UI通常继承自UIBehaviour，它会提供一些方便的函数如果打开UIBehaviour类的话，会发现其实它也是继承自MonoBehaviour的。

GetModifiedMaterial函数中的内容非常简单，首先将之前函数的内容复制进来：

Mymateraial.CopyPropertiesFromMaterial(baseMaterial);

接着去set我们想要修改 参数即可。

此外，我们需要检测UI各处的变化，并在变化时调用SetMaterialDirty()

如下：

OnEnable 在组件变为启用状态并且所在 GameObject 处于激活状态时触发。

 protected override void OnEnable()
 {
     base.OnEnable();
     if (graphic == null) return;
     _image = graphic as Image;
     _graphic.SetMaterialDirty();
 }

OnDisable 在组件失效时触发。

protected override void OnDisable()
{
    base.OnDisable();
    if (material != null)
    {
        DestroyMaterial();
    }
    if (graphic != null) graphic.SetMaterialDirty();

}

OnValidate 主要是编辑器里触发的校验回调。

protected override void OnValidate()
{
    base.OnValidate();
    if (!IsActive() || graphic == null) return;
    graphic.SetMaterialDirty();
}

OnDidApplyAnimationProperties()会在动画系统把动画属性应用到该对象后触发。

protected override void OnDidApplyAnimationProperties()
{
    base.OnDidApplyAnimationProperties();
    if (!IsActive() || graphic == null) return;
    graphic.SetMaterialDirty();
}

以上是正常情况下的检测状况，足以应对大部分UI事件

但是如果有每帧修改的需求的话，也可以在Update中SetMaterialDirty

void Update()
{
    if (!IsActive() || graphic == null) return;
    graphic.SetMaterialDirty();
}
