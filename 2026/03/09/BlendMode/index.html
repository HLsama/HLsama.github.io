BlendMode

2026/3/9 技术 UI

混合模式本身也并不是什么很难的技术，因此其本身并没有什么好讲的，但是其中涉及到Unity的透明剪切与透明度混合等内容是非常值得一说的，这将会是本文的重点。

unity版本：Unity6000.062f1

1.Unity的渲染流程

众所周知，渲染的本质就是将一堆顶点组成的三角形经过各种变换，最后变换到屏幕空间，然后再去判断这些三角形覆盖哪些像素，最后在像素着色器中计算出每个像素的颜色值。而这其实并没完，由于多个三角形可能覆盖同一个像素，所以我们需要某些方式来决定这些像素的去留与混合方式。根据先后执行的方式，他们分别是：模板测试，深度测试，透明剪切，透明度混合。通常物体的话通常会有个渲染顺序，先渲染天空盒，然后是不透明物体，半透明物体，最后是UI。然后不透明物体的渲染顺序是从前到后，而半透明是从后到前。这些规则保证了正常的物体遮挡关系。而它又和透明度混合密切相关，不过由于本文重点是UI，所以对于通常物体渲染就不多介绍了，后文会更多的介绍UI的透明度混合部分。

2.UI的透明度混合

如果没有特殊情况，UI是在通常物体之后渲染的，这也意味着UI近乎永远在物体的最前面，他会遮挡住物体。而UI之间呢虽然都在同一个平面，但他们之间是有渲染顺序关系的，通常在同一个Canvas 中Hierarchy 里越靠后的对象，越晚渲染，显示得越靠上，在半透明渲染中渲染顺序相当重要，通常越靠前的物体它的渲染顺序越靠后，这样才能保证物体的透明度混合是正确的。造成结果差异的关键就是透明度混合的公式：

对于任何像素，我们有像色着色器中输出的颜色，我们称之为源颜色（Src），而颜色缓冲中的颜色，也就是屏幕上已有、待混合的颜色，我们称之为目标颜色（Dst），而透明度混合就是：Src x _SrcFactor _BlendOp Dst x _DstFactor。其中_BlendOp是混合运算，会决定如何将前后两个式子叠加：

而SrcFactor 与DstFactor决定源和目标预先进行怎样的乘法。通常的式子是SrcAlpha、OneMinusSrcAlpha，也就是源乘以源的透明度，目标乘以一减去源透明度，这也解释了渲染顺序的不同为何导致渲染结果的不同。而这个乘上去的因子是不固定的，可以根据需求自由搭配，下面是unity官方的这些混合因子的列表：

One 值为 1 - 让源或目标颜色通过。
Zero 值为 0 - 删除源或目标值。
SrcColor 此阶段的值乘以源颜色值。
SrcAlpha 此阶段的值乘以源 Alpha 值。
DstColor 此阶段的值乘以帧缓冲区源颜色值。
DstAlpha 此阶段的值乘以帧缓冲区源 Alpha 值。
OneMinusSrcColor 此阶段的值乘以（1 - 源颜色）。
OneMinusSrcAlpha 此阶段的值乘以（1 - 源 Alpha）。
OneMinusDstColor 此阶段的值乘以（1 - 目标颜色）。
OneMinusDstAlpha 此阶段的值乘以（1 - 目标 Alpha）。

以及一些详细的混合类型：

  • Blend SrcAlpha OneMinusSrcAlpha // 传统透明度
  • Blend One OneMinusSrcAlpha // 预乘透明度
  • Blend One One // 加法
  • Blend OneMinusDstColor One // 软加法
  • Blend DstColor Zero // 乘法
  • Blend DstColor SrcColor // 2x 乘法

其中第一种就是unity对于半透明物体使用的混合方式，而UI一般会预先乘Alpha，所以一般使用第二种。

更多内容可以看一下unity官方关于透明度混合的文档：Unity - Manual: Blend command in ShaderLab reference
